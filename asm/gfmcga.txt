
Refresh_Dirty_Tiles
Description:
Main tile refresh routine. Marks the tile cache as dirty for all tiles, then iterates over the 28×19 viewport tilemap, re-rendering any tile that has changed (dirty flag) or is animated. It also calls special handlers for the top‑left and bottom‑right corner entities, and for tiles that require animation updates based on cavern level.

Input:

ds:viewport_left_top_addr – pointer to the start of the current viewport in the tilemap.

ds:viewport_top_left_vram_addr – VRAM address of the top‑left corner of the viewport.

ds:viewport_buffer_28x19 – buffer holding the currently displayed tile indices.

ds:roka_palette – animation frame counter (incremented by the routine).

ds:plane0_buf – set to viewport VRAM base address.

Various other global state (cavern level, etc.).

Output:

VRAM segment A000h is updated with dirty tiles, animations, and sprites.

ds:viewport_buffer_28x19 is updated with new tile indices where changed.

ds:viewport_rows_remaining is decremented per row.

ds:roka_palette is incremented once.
====================================
Sample_Neighborhood_Attributes
Description:
Loads the 3×3 block of tile indices around the hero’s current position from the proximity map and stores them into tile_neighborhood_buffer. Used later to determine what tiles are under or near the hero for proper rendering and attribute lookups.

Input:

ds:hero_y_absolute – hero’s absolute Y coordinate.

ds:hero_x_in_viewport – hero’s X offset within the viewport.

ds:viewport_left_top_addr – base address of the viewport tilemap.

Proximity map at proximity_map.

Output:

tile_neighborhood_buffer (9 bytes) filled with tile indices (negative values indicate valid loaded tiles, zero if blank).
====================================
Flush_Ui_Element_If_Dirty
Description:
If the UI element (e.g., the hero’s upper body displayed outside the scrolling view) is marked dirty, this routine blits a 32×32 sprite from the temporary buffer at entity_vram_src to VRAM at address 64064 (a fixed screen location). It then clears the dirty flag.

Input:

ds:ui_element_dirty – non‑zero if the UI element needs updating.

cs:entity_vram_src – pointer to the 32×32 sprite data in shadow VRAM.

VRAM segment A000h.

Output:

32×32 region on screen updated.

ds:ui_element_dirty cleared to 0.
====================================
Render_Scrolling_Transition_Overlay
Description:
Handles the sword‑swing overlay during scrolling transitions. Based on sword_swing_flag, it selects the appropriate sword animation graphics and sets up the sprite offset and source data for later rendering. When the sword animation sequence ends, it resets the flags.

Input:

ds:sword_swing_flag – if 0, no sword animation; else triggers overlay.

ds:sword_hit_type – type of sword swing (0, 1, or other).

ds:sword_movement_phase – current phase of the sword animation.

ds:facing_direction – hero facing direction (bit0: 0=right, 1=left).

ds:squat_flag – non‑zero if hero is squatting.

Output:

ds:hero_sprite_offset set to (X, Y) offset for the sword sprite.

ds:entity_vram_src set to the hero’s VRAM base adjusted by offset.

ds:sword_phase_src set to pointer within sword_animation_gfx for the current phase.

ds:sword_movement_phase incremented; on final phase, sword_swing_flag and sword_movement_phase reset to 0.
====================================
Uncompress_And_Render_Tile
Description:
Decodes a 48‑byte packed tile from seg1:8030h (index given in AL) and draws it directly to VRAM at the screen address provided in DI (half of actual address). The tile data is unpacked 3 bytes → 4 pixels, with zeros skipped to improve performance.

Input:

AL – tile index (1‑based; 0 means blank).

DI – half of the VRAM destination address (actual address = DI * 2).

ds must point to segment seg1 where packed tiles are stored.

Output:

8×8 tile rendered at es:di*2 in VRAM.

DI advanced by tile width (8) and row stride (320) as needed.
====================================
Viewport_Coords_To_Screen_Addr
Description:
Converts viewport pixel coordinates (X in AH, Y in AL) to a VRAM address offset (returned in DI) for the 320×200 MCGA screen. The viewport is assumed to start at screen position (14*320+48), i.e., a small border around the game area.

Input:

AL – Y coordinate (0‑63, but only 0‑63 used; masked to 0‑63).

AH – X coordinate (4‑31, shifted to be relative to viewport left edge, subtract 4).

Output:

DI – half of the actual VRAM address (address = DI * 2). The caller usually doubles it to get the final address.
====================================
Scrolling_Overlay_EntryPoint
Description:
Entry point for rendering the sword overlay on top of the hero during a sword swing. It clears the tile cache around the hero, copies the hero’s upper‑half background to a temporary buffer, then draws the sword animation tiles using sprite blitting with transparency.

Input:

ds:sword_swing_flag – must be non‑zero.

ds:sword_type, ds:facing_direction, ds:squat_flag, etc. – determine which sword graphics to use.

cs:entity_vram_src and cs:sword_phase_src – prepared by Render_Scrolling_Transition_Overlay.

Output:

VRAM updated with the sword sprite over the hero’s upper body.

ds:ui_element_dirty set to 0FFh (marks UI element dirty for later flush).

Tile cache entries around hero invalidated.
====================================
Cached_Tile_Drawer
Description:
Draws a single 8×8 tile using a cache to avoid repeated decompression. If the tile is already cached in shadow VRAM (tile_vram_cache), it is blitted quickly using movsw. If not cached, the tile is decompressed (3‑bytes‑to‑4‑pixels) from seg1:8030h and both written to VRAM and cached. Tile index 0 is treated as a clear (all zeros) tile.

Input:

AL – tile index (0 = clear, 1‑based otherwise).

DX – VRAM destination address divided by 2 (actual address = DX*2).

Output:

Tile rendered at VRAM address DX*2.

tile_vram_cache updated for the given index.
====================================
Active_Entity_Sprite_Renderer
Description:
Iterates through a list of active map entities (max 32) and renders each one as a 16×16 sprite onto the viewport. Entities that have expired (flag 0FFh) are removed. Each entity is drawn using a mask table and an entity‑render‑function table that defines the transparency bitplane.

Input:

active_entity_table – array of entities (x, y, life counter, type, etc.).

viewport_buffer_28x19 – to mark tiles under entity as “dirty” (so they are redrawn later).

ds:entity_render_fn_tbl and ds:entity_mask_tbl – lookup tables for rendering.

Output:

Entities drawn to VRAM; expired ones removed.

active_entity_table updated.
====================================
Render_Viewport_Tiles
Description:
Performs a full redraw of the 28×19 tile viewport eight times, each with a different animation frame (frame counter from 8 down to 1). It iterates over every tile in the viewport, calling render_tile for each. This routine is used for the “Roca” area or similar full‑screen tilemap display.

Input:

ds:viewport_left_top_addr, ds:viewport_buffer_28x19.

ds:plane0_buf set to viewport VRAM base.

ds:roka_palette initially set to 8.

Output:

Entire 28×19 tile area rendered eight times (for animation).

Frame timer (ds:frame_timer) used for pacing.
====================================
Copy_Tile_Buffer_To_VRAM
Description:
Copies the 3×3 tile buffer of the hero (stored at 511Dh in shadow VRAM) to the screen at the hero’s current VRAM position (hero_vram_base). Used to transfer the pre‑rendered hero sprite tiles to the visible display. If the hero sprite is hidden (hero_sprite_hidden set), the copy is skipped.

Input:

ds:hero_sprite_hidden – flag.

cs:hero_vram_base – destination VRAM address.

Shadow VRAM at 511Dh contains the 3×3 tiles.

Output:

3×3 block of tiles copied to screen VRAM.

ds:hero_sprite_hidden set to 0FFh after copying.
====================================
Update_Local_Attribute_Cache
Description:
Clears the tile neighborhood buffer and then calls Sample_Neighborhood_Attributes to fill it. Then, for each of the 3×3 tiles under the hero, it determines which sprite (body, arms, shield, etc.) to render based on the hero’s state (facing, squat, shield, rope, invincibility, etc.). It prepares the tile indices and then calls Render_Scrolling_Tile or Render_Tile_With_Palette for each tile. This is the main hero‑rendering routine.

Input:

All hero state variables (position, facing, animation phase, shield, squat, etc.).

ds:hero_x_in_viewport, ds:hero_head_y_in_viewport, etc.

tile_neighborhood_buffer filled by Sample_Neighborhood_Attributes.

Output:

Shadow VRAM at sword_anim_frames (or equivalent) filled with the hero’s 3×3 tile graphics.

ds:hero_vram_base computed from hero coordinates.

Global tile cache may be updated.
====================================
Render_Viewport_Border_Walls
Description:
Draws the animated border walls around the viewport (the “rings” of tiles that enclose the play area). It uses a dithered pattern and then calls RenderBorderRings twice with different tile animation frame counters to create a moving effect.

Input:

ds:hero_x_in_viewport and ds:hero_head_y_in_viewport – used to seed the border pattern.

Output:

Border walls rendered to VRAM around the viewport.

Dither pattern applied.
====================================
Load_Magic_Spell_Sprite_Group
Description:
Loads the sprite sheet for the currently selected magic spell (1‑6) from segment seg2:0 into a buffer at seg1:9350h. If no spell is selected (0 or 7), nothing is loaded.

Input:

ds:current_magic_spell – spell index (1‑6).

cs:seg2 – segment containing compressed spell graphics.

cs:seg1 – target segment for uncompressed graphics.

Output:

seg1:9350h filled with spell sprite data (max 480h bytes).

ds:si set to 9350h.
====================================
Render_Animated_Tile_Strip
Description:
Renders a 5×28 block of animated tiles (such as the sword‑hit effect) using a set of tile indices from sword_hit_pattern_tbl. Each tile is drawn from the compressed tile data at seg1:4000h using a fixed transparency mask (1210h).

Input:

cs:transparency_mask_bitplane_f set to 1210h.

si points to sword_hit_pattern_tbl (array of tile indices).

di initial VRAM destination 48+(14+26)*320.

Output:

Animated strip drawn to VRAM.
====================================
Render_Roca_Tilemap
Description:
Renders the 28×18 tilemap for the “Roca” area (a special location). It iterates over the predefined roca_tile_indices_28x18 array and draws each tile using RenderTileFrom_seg1. The tile animation frame counter is provided in AL.

Input:

AL – tile animation frame counter.

roca_tile_indices_28x18 – array of tile indices.

ds:plane0_buf – set to viewport VRAM base.

Output:

28×18 tile region rendered to VRAM.
====================================
Calculate_Tile_VRAM_Address
Description:
Computes the VRAM address for a 16×16 sprite given its X and Y tile coordinates. It stores the result in ds:hero_vram_base and then falls through to the next routine (Copy_Tile_Buffer_To_VRAM). Input registers: AH = Y tile coordinate, BH = X tile coordinate.

Input:

AH – Y tile coordinate.

BH – X tile coordinate.

Output:

ds:hero_vram_base = VRAM address of the tile position.

Falls through to Copy_Tile_Buffer_To_VRAM.
====================================
Render_16x16_Sprite
Description:
Draws a 16×16 sprite from the tile graphics buffer at seg1:6000h (compressed tile data). The sprite is made of 8×8 tiles, but this routine reads compressed tile rows (each row 2 bytes plane0, 2 bytes plane1) and unpacks them into 4 mask bytes per group.

Input:

AH – Y coordinate (tile offset).

BH – X coordinate (tile offset).

AL – tile index? (used to compute offset into 6000h; cl = 32, mul cl → offset into sprite data).

ds points to seg1.

Output:

16×16 sprite rendered at the given VRAM coordinates.

Uses Calculate_Tile_VRAM_Address logic for address calculation.
====================================
Render_Status_Indicator
Description:
Renders a status indicator (e.g., a health bar or similar) using a pre‑defined mask table. It selects the appropriate mask based on byte ds:92h (likely a status value) and draws a 24×8 (or 24×16?) sprite using the CalculateSpriteBitmask routine.

Input:

ds:92h – status indicator index (1‑?).

cs:transparency_mask_bitplane_f set to 908h.

di initial VRAM address 6C10h.

Output:

Status indicator drawn on screen.
====================================
Render_Entity_Sprite
Description:
Renders a single game entity sprite (e.g., monster or NPC). Depending on the sign of AL (negative means left‑facing, positive means right‑facing), it selects a base sprite address and a width multiplier (1 column for right, 4 columns for left). It then draws the sprite column by column using CalculateSpriteBitmask.

Input:

AL – sprite orientation/frame index (sign determines direction).

CX – Y coordinate in pixels? (used as row multiplier).

BX – X coordinate in pixels? (added to computed offset).

ds – data segment containing sprite data tables (sprite_data_base_right, sprite_data_base_left).

Output:

Entity sprite drawn at the computed VRAM address.
====================================
Decompress_Tile_Data
Description:
Decompresses a set of 8×8 tile graphics from a custom 2‑plane nibble‑packed format into a standard 8‑bytes‑per‑row format suitable for the MCGA screen. It reads compressed data (32 bytes per tile) from DS:SI, copies it to a temporary buffer in seg3, and unpacks each row, building two 16‑bit bitplanes and generating transparency masks. The output is stored at ES:DI (destination) and the transparency mask at ES:BP. The number of tiles is given in CX.

Input:

DS:SI – pointer to compressed tile data (32 bytes per tile).

ES:DI – destination for decompressed pixel data.

BP – destination for transparency mask bytes.

CX – number of tiles to decompress.

cs:plane0_buf, cs:plane1_buf – temporary storage.

Output:

ES:DI filled with decompressed tile rows (each row is four 16‑bit words).

ES:BP filled with transparency mask bytes (1 per row per tile).

Registers may be modified.
====================================
nullsub_1
Description:
A placeholder null subroutine that simply returns. Likely used as a dummy entry in a jump table.

Input/Output: None.
------------------------------------
Internal Helper Functions
For completeness, here are annotations for the smaller helper routines that are not directly exposed in the main jump table but are part of the code.
------------------------------------
Process_Dirty_Tile_With_Animation
Handles a tile that differs from the cache. If the tile is animated, it increments the cache flag, and if necessary, redraws it via Cached_Tile_Drawer. For certain cavern levels (5‑8), it calls animation‑specific subroutines (Animate_Door_Tile_Type0, etc.) to update the tile index.
====================================
Animate_Door_Tile_Type0
Animates door tiles (indices 1Bh‑1Ch) by toggling between the two frames every other animation tick.

Animate_Chest_Lever_Tile
Animates chest/lever tiles (indices 1Dh‑22h). Chests cycle through four frames; levers cycle through two frames.

Animate_Special_Tiles
Animates various special tiles (indices 2Ch‑3Dh) by swapping to a partner tile index every other animation tick.

AnimateWaterTile
Animates water tiles (indices 25h‑28h) by cycling through four frames.

Render_Top_Left_Corner_Entity
Renders the entity that overlaps the top‑left corner of the viewport. It uses the tile index from the tilemap, looks up its attributes via Lookup_Monster_Tile_Attributes, and calls Decode_And_Render_Tile_With_Blit.

Render_Tile_With_Attribute_Cache
Renders a single 8×8 tile that is part of the viewport, taking into account neighboring tiles and caching. It loads the 3×3 tile neighborhood, looks up monster tile attributes, and calls Render_Tile_Neighborhood_Cell for each of the two rows.

Render_Bottom_Right_Corner_Entity
Similar to top‑left corner, but for the bottom‑right corner of the viewport.

Render_Tile_With_Dual_Cache
Handles tiles that span two cache rows (e.g., when the hero crosses a tile boundary). It checks dirty flags for both the current and next row, and if not fully cached, decodes and renders the tile.

Render_Tile_With_Border_Check
Renders a tile while also checking if it is on the border of the viewport and may need to spawn map entities (in boss caverns with sprite flash). It also updates the hero’s tile column/row for entity spawning.

Render_Tile_And_Update_Cache
Renders a tile and updates the cache, used when the tile is not already dirty. It handles both the current and next row of the tile neighborhood.

Render_Tile_Neighborhood_Cell
A helper that renders one cell of a 2×2 tile group (top‑left, top‑right, bottom‑left, bottom‑right) by calling Decode_And_Render_Tile_With_Blit if the cache flag indicates it is dirty.

Decode_And_Render_Tile_With_Blit
Main routine to decompress a nibble‑compressed tile and render it either with or without a blit (transparency) mode. It uses the current tile_blit_mode to select a palette decode table (pal_decode_tbl), decompresses the tile from seg1:4000h, optionally blends with the background using a mask, and writes it to VRAM.

decode_and_render_tile_with_blitting
Renders a tile by first decompressing the background tile from packed 48‑byte format (using render_48bytes_packed_tile) and then overlaying a hero tile (from nibble‑compressed data) using transparency masks (render_2bpp_tile). Used for drawing the hero over the background.

four_pixels_or_blit
Processes 4 pixels at once: extracts a blit mask from DL (high bits) and a pixel colour from AX (high nibbles) using the LUT, then combines them with the background using AND‑OR logic.

render_nibble_compressed_tile
Draws an 8×8 tile directly from nibble‑compressed data (32 bytes, 64 nibbles) to VRAM. Each nibble indexes a 16‑entry colour LUT.

draw_4_pix_from_table_by_ax
Helper for render_nibble_compressed_tile: writes 4 pixels using the current LUT.

get_pixel_from_table_by_ah_hi
Extracts the high nibble of AH and uses it as an index into cs:nibble_decode_lut to get a pixel colour byte.

Copy_tile
Copies an 8×8 tile from DS:SI to ES:DI with a row stride of 320 bytes (MCGA screen pitch).

render_48bytes_packed_tile
Unpacks a 48‑byte per tile format (3 bytes → 4 pixels) directly to VRAM. Used as a fast way to render pre‑decompressed tile data.

Clear_Tile_Buffer
Clears the 32‑byte tile buffer at ES:DI (sets it to zero). Used for empty tiles.

Translate_Tile_Index
Translates a raw tile index by looking it up in the proximity_second_layer table. Used to map “second layer” tiles (e.g., those overlaid on the hero) to their actual graphics indices.

Lookup_Monster_Tile_Attributes
Given a tile index (in AL), looks up the associated monster tile attributes from the monsters_table_addr. Returns the blit mode in ds:tile_blit_mode and the translated tile index in AL. Also accounts for boss caverns and special flags.

Spawn_Map_Entity
Randomly spawns a small map entity (like a wandering creature) in the viewport area if conditions are met (boss cavern, sprite flash flag set, and random chance). Uses active_entity_table.

choose_hero_sprite
Determines which 9 tiles (or 6 for squatting) form the hero’s body, arms, shield, etc., based on the hero’s state (facing, animation phase, shield, squat, jump, slope, invincibility, rope). It then calls Render_Scrolling_Tile to draw each tile.

Render_Scrolling_Tile
Renders a series of tiles (CX count) from the sprite sheet at ES:SI, applying the hero’s transparency mask. Each tile is decompressed and blitted over the background. This is used to draw the hero’s body parts one by one.

get_player_shield_category
Returns the shield size category in AL: 0 = no shield, 1 = small shield (Clay, WiseMan’s, Stone), 2 = large shield (Honor, Light, Titanium). Input from ds:shield_type.

Render_Empty_Or_Cached_Tile
Draws a single 8×8 tile at the hero’s current position, either from the packed tile cache (48‑byte) or, if tile index is zero, clears the buffer. Used during hero rendering to fill background tiles under the hero.

Render_Tile_With_Palette
Renders a tile using the current blit mode palette, blending with the background via decode_and_render_tile_with_blitting if the tile is non‑zero, or clearing the tile buffer if zero.

load_3x3_tiles
Loads the 3×3 tile indices surrounding the hero from the viewport tilemap into tile_neighborhood_buffer.

render_hero_sword
Checks if the hero is swinging his sword and, if so, performs the necessary UI element flush and overlay rendering via Scrolling_Overlay_EntryPoint.

Flush_Ui_Element_If_Dirty (duplicate entry in the list)
Already described.

Blit32x32SpriteToVram
Blits a 32×32 sprite from shadow VRAM (at cs:entity_vram_src) to the actual VRAM at the same address, effectively making the pre‑rendered hero UI visible.

CopyTempBufferToVRAM_RowMajor
Copies a 32×32 region from cs:entity_vram_src (the hero’s upper half) to a temporary buffer at VRAM address 64064 using row‑major order.

Clear_Tile_Cache_Around_Hero
Invalidates the tile cache for the 4×4 tile area around the hero. This forces those tiles to be re‑rendered later, ensuring no stale data when the hero moves.

CalculateSpriteBitmask
Given a 4‑bit pixel pattern in AX (bits 15‑12), generates a bitmask (BP) and a colour value (DX) based on the current transparency_mask_bitplane_f. Used for sprite blitting.

Render_Viewport_Tiles (already described)
render_tile
Core routine to render a single tile within Render_Viewport_Tiles. It checks the dirty cache flag, and if dirty, decompresses and draws the tile in two passes (using the animation frame counter to select sub‑tile rows).

RenderTileRowWithMask / ClearTileRowWithMask / CalculateTileOffset
Helpers for the animated tile rendering in render_tile. RenderTileRowWithMask draws one row of 4 pixels from the 48‑byte packed tile, applying the tile render mask. ClearTileRowWithMask clears a row (sets it to background). CalculateTileOffset adjusts SI and DI based on the animation frame.

Render_Viewport_Border_Walls (already described)
RenderBorderRings / RenderBorderSegment / RenderOrthogonalSegments / DrawVerticalLine / DrawHorizontalLine / CalculateRowVRAMAddress
These routines work together to draw the animated border “rings”. They draw horizontal and vertical lines using a dither pattern and the tile animation counter.

WaitForVBlankAndDelay
Waits for a frame to complete and handles speed‑change, pause, joystick, and exit dialog callbacks.

DrawDitheredPattern
Applies a dither pattern (1001000010010b) over the viewport background (likely the space outside the play area) to create a visual effect.

Load_Magic_Spell_Sprite_Group (already described)
wrap_e900_from_above / wrap_e000_from_below
Pointer wrapping helpers for the viewport tilemap, which is stored in a 0x900‑byte circular buffer.

Render_Animated_Tile_Strip (already described)
Render_Roca_Tilemap (already described)
RenderTileFrom_seg1
Draws a tile from the packed tile data in seg1 (48 bytes per tile) using Render8pxWithPaletteTransform, which also applies a palette transform based on the tile animation counter.

Render8pxWithPaletteTransform
Unpacks one row (8 pixels) and then optionally recolours each pixel using one of five PaletteTransform_* routines, allowing colour cycling effects.

PaletteTransform_0 to 4
Small lookup tables that remap colour indices for special animation effects (e.g., palette cycling in the Roca area).

Render_16x16_Sprite (already described)
Unpack_4MaskBytes
Unpacks a 16‑bit word (from DX) into four bytes of mask data for sprite rendering.

Render_Status_Indicator (already described)
Render_Entity_Sprite (already described)
Decompress_Tile_Data (already described)
build_16_bits_from_2_planes
Rotates two plane buffers and builds a 16‑bit value that combines 4 bits from each plane.

extract_transparency_byte_from_mask_plane_f
Extracts an 8‑bit transparency mask from the transparency_mask_bitplane_f word, generating a byte where a bit is 1 only if two consecutive bits in the mask are both set.

These annotations cover every procedure listed in the code snippet. The driver uses a combination of direct VRAM writes, tile caching, nibble decompression, and palette‑based blitting to achieve the real‑time graphics of Zeliard on MCGA hardware.
